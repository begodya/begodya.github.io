CSS之BFC、IFC、GFC和FFC

发表于 2019-03-25

BFC

以<div></div>为例，<div></div>被浏览器解析后会生成div元素并添加到DOM树中，CSS作用的对象不是DOM树，而是根据DOM树生成的Render树，BFC就是Render树的节点。

方法
  • float：不为none
  • position：不为relative、static
  • overflow：不为hidden
  • display：table-cell、table-caption、inline-block
结构
  • content box（必备）
  • padding box
  • border box
  • margin box
作用
  • 清除浮动
  • 自适应两栏布局
  • 防止margin合并
注意
  • CSS使用于Box，而不是作用于Element
  • JS无法操作Box
  • 同一个BFC的两个Box，margin会有重叠
误区
  • 通过width/height属性设置div元素的宽/高（错误）
  • CSS属性width/height作用于div元素所产生的Box，而不是Element本身
  • BFC有4个Box组成，那width/height到底作用于哪个Box？

IFC

因素
  • font-size
  • line-height
  • height
  • vertical-aligin
注意
  • inline-level box的高度是由字体字号大小决定的
  • inline-level box的宽度是由content+padding+border+margin决定的
  • inline-level box宽度大于父容器的宽度时会被拆分成多个inline-level box
  • 当达到拆分条件时，inline-level会被拆分成多个inline-level box并分布到多行中。
    • 当direction为ltr时，margin/border/padding-left只会作用于第一行inline-level box；margin/border/padding-right作用于最后一行inline-level box。
    • 当direction为rtl时，margin/border/padding-right只会作用于第一行inline-level box；margin/border/padding-left作用于最后一行inline-level box。

GFC

  • display: grid;

FFC

  • display: flex;
  • display: inline-flex;
布局实例
  • Flex 布局教程：语法篇
  • Flex 布局示例
注意
  • 设为 Flex 布局以后，子元素的float、clear和vertical-align属性将失效

React Navigation 3x开发指南

发表于 2019-01-30

什么是导航器？

App基本不会只由一个页面组成，管理多个页面的呈现、跳转的组件就是导航器。

React Navigation 3x中6种类型的导航器

  1. createStackNavigator：屏幕上方导航栏，iOS右侧滑入（可配置底部滑入），Android底部淡入；
  2. createBottomTabNavigator：屏幕下方标签栏；
  3. createMaterialBottomTabNavigator：屏幕底部的材料设计主题标签栏；
  4. createMaterialTopTabNavigator：屏幕顶部的材料设计主题标签栏，可通过点击或水平滑动来切换；
  5. createDrawerNavigator：抽屉效果，侧边滑出；
  6. createSwitchNavigator：一次只显示一个页面；

可以通过以上6种导航器来创建App，根据应用场景来使用其中一个或是多个组合

Navigation prop reference

应用中每个页面组件都会自动提供navigation prop，包含以下方法用于触发导航操作：

  • navigate：跳转到另一个页面
  • goBack：关闭当前页面
  • dispatch：向路由发送 action
  • state：当前的状态或路由
  • setParams：对路由参数进行更改
  • getParam：获取具有回退的特定参数
  • addListener：订阅导航生命周期的更新
  • isFocused：屏幕是否有焦点
  • dangerouslyGetParent：返回父级导航器
StactNavigator中附加方法
  • this.props.navigation
    • push：推一个新的路由到堆栈
    • pop：返回到堆栈中的上一个页面
    • popToTop：返回到堆栈中最顶层的页面
    • replace：用新路由替换当前路由
    • reset：清除整个导航状态，并将其替换为多个操作的结果
    • dismiss：关闭当前堆栈
DrawerNavigator中附加方法
  • this.props.navigation
    • openDrawer：打开
    • closeDrawer：关闭
    • toggleDrawer：切换

通用API使用方法

使用navigate进行页面跳转
  • navigation.navigate({ routeName, params, action, key})
  • navigation.navigate( routeName, params, action)
    • routeName：要跳转到的页面的路由名，即导航中配置的路由名
    • params：传递给下一个页面的参数
    • action：如果页面是navigator，则是子路由中运行的子操作
    • key：要导航到的路由的可选标识符。如果已存在，将后退到此路由
使用goBack进行关闭当前页面并返回上一个页面
  • goBack(key)
    • key：表示要返回到的页面标识，而不是routeName
    • 可以通过navigation.state.key来获取页面标识
    • key非必传，也可为null
使用dispatch进行向路由发送action
  • dispatch(action)：给当前界面设置action，会替换原来的跳转，回退等事件
使用state获取当前的state或route
  • 页面可以通过this.props.navigation.state访问其路由
使用setParams对路由的参数进行更改
  • setParams(params)：触发setParams方法允许页面更改路由中的参数，可以更新当前页面的标签和按钮
使用getParam获取指定的参数，可设置获取失败的返回值
使用addListener订阅导航生命周期的更新
  • React Navigation将事件发送到订阅了它们的页面组件：
    • onWillFocus：页面将获取焦点
    • onDidFocus：页面已获取焦点
    • onWillBlur：页面将失去焦点
    • onDidBlur：页面已失去焦点
使用isFocused进行查询页面是否获取到焦点
  • let isFocused = this.props.navigation.isFocused() * 如果页面已获取焦点，返回true，否则返回false
使用dangerouslyGetParent进行获取父级导航

客户端跨平台解决方案

发表于 2018-12-13

目前有许多跨平台UI框架的方案，比如基于WebView的Cordova、AppCan等，还有使用HTML+JavaScript渲染成原生控件的React Native、Weex等。

第一阶段：基于WebView的框架优点很明显，它们几乎可以完全继承现代Web开发的所有成果（丰富得多的控件库、满足各种需求的页面框架、完全的动态化、自动化测试工具等等），当然也包括Web开发人员，不需要太多的学习和迁移成本就可以开发一个App。同时WebView框架也有一个致命（在对体验&性能有较高要求的情况下）的缺点，那就是WebView的渲染效率和JavaScript执行性能太差。再加上Android各个系统版本和设备厂商的定制，很难保证所在所有设备上都能提供一致的体验。

第二阶段：为了解决WebView性能差的问题，以React Native为代表的一类框架将最终渲染工作交还给了系统，虽然同样使用类HTML+JS的UI构建逻辑，但是最终会生成对应的自定义原生控件，以充分利用原生控件相对于WebView的较高的绘制效率。与此同时这种策略也将框架本身和App开发者绑在了系统的控件系统上，不仅框架本身需要处理大量平台相关的逻辑，随着系统版本变化和API的变化，开发者可能也需要处理不同平台的差异，甚至有些特性只能在部分平台上实现，这样框架的跨平台特性就会大打折扣。

第三阶段：Flutter开辟了一种全新的思路，从头到尾重写一套跨平台的UI框架，包括UI控件、渲染逻辑甚至开发语言。渲染引擎依靠跨平台的Skia图形库来实现，依赖系统的只有图形绘制相关的接口，可以在最大程度上保证不同平台、不同设备的体验一致性，逻辑处理使用支持AOT的Dart语言，执行效率也比JavaScript高得多。

前两阶段是否真正的解决了跨平台的问题？从目前的状况来看，很显然是没有的，因为它们始终都逃不开性能、包大小、内存、流畅性，平台特性等问题。

React Native

React Native 的口号是”Learn once, write anywhere”。

React Native (简称RN)是Facebook于2015年4月开源的跨平台移动应用开发框架，是Facebook早先开源的JS框架 React 在原生移动应用平台的衍生产物，目前支持iOS和安卓两大平台。RN使用Javascript语言，类似于HTML的JSX，以及CSS来开发移动应用，因此熟悉Web前端开发的技术人员只需很少的学习就可以进入移动应用开发领域。

目前存在的问题

  1. 全新的技术栈React学习
  2. 严重依赖Fackbook的维护
  3. 当前版本是0.57，仅支持简单的UI制作，无法满足复杂的动画实现，更别提iOS底层的优化和兼容操作
  4. 联调困难

Weex

Weex 的口号是”Write once, run everywhere”。

Weex 是一个使用 Web 开发体验来开发高性能原生应用的框架。Weex 致力于使开发者能基于当代先进的 Web 开发技术，使用同一套代码来构建 Android、iOS 和 Web 应用。具体来讲，在集成了 WeexSDK 之后，你可以使用 JavaScript 和现代流行的前端框架来开发移动应用。

Weex 的结构是解耦的，渲染引擎与语法层是分开的，也不依赖任何特定的前端框架，目前主要支持 Vue.js 和 Rax 这两个前端框架。

Weex 的另一个主要目标是跟进当代先进的 Web 开发和原生开发的技术，使生产力和性能共存。在开发 Weex 页面就像开发普通网页一样；在渲染 Weex 页面时和渲染原生页面一样。

目前存在的问题

  1. Environment variable $ANDROID_HOME not found !
  2. No toolchains found in the NDK toolchains folder for ABI with prefix: arm-linux-androideabi
  3. 单页面结构与多页面结构
  4. 页面跳转
  5. 组件太少
  6. 不支持部分CSS属性
  7. 兼容性问题
  8. 开发者需要精通三端框架，可以处理组件无法满足需求时进行原生定制
  9. Swift/OC、Kotlin/Java、Hybrid、Weex混合编译问题目前无法预知
  10. 无社区问题，遇到问题无法及时响应处理
  11. 文档是硬伤

Flutter

Flutter 理念是一切皆为Widget。

Flutter 跨平台的最核心部分，是它的高性能渲染引擎。Flutter 不使用浏览器技术，也不使用Native的原生控件，它使用自己的渲染引擎来绘制Widget。Widget 是 Flutter 应用程序用户界面的基本构建块，每个 Widget 都是用户界面一部分的不可变声明。对于 Android 平台，Flutter 引擎的 C/C++ 代码是由 NDK 编译；在iOS平台，则是由LLVM编译，两个平台的Dart代码都是AOT编译为本地代码，Flutter 应用程序使用本机指令集运行。

Flutter 通过使用相同的渲染器、框架和一组 Widget，来同时构建 iOS 和 Android 应用，而无需维护两套独立的代码库。

目前存在的问题

  1. 全新的技术栈Dart学习
  2. 生态不完善，相关资料较少
  3. 缺少混合开发文档

总结

单从技术角度来说，这几种跨平台方案都是不可多得的优秀框架。但脱离应用谈技术，就好比是镜中月、水中花，不切合实际。实际App开发中，有几种情况来分析下是否适合使用跨平台：

第一种：从零开发一款App，这种情况还需要看当前团队中的人员配置情况和业务逻辑

  • 如果没有Native的技术栈人员或是且业务相对简单，非常适合引入跨平台方案，快速上线占领市场。
  • 如果有Native的技术栈人员且业务相对复杂，有可能是公司新的产品线，或是成熟的App进行重构，考虑到产品的质量、口碑、及日后的维护，Native对实际问题的解决方案更加成熟。

第二种：在原有的App中引入新功能

  • 原有App代码库是100% Native，原有技术栈已经成熟，引入第三个技术栈后会增加代码的维护成本和复杂度。除非后期有大幅采用跨平台的计划，及跨平台转队，否则不建议引入跨平台方案。
  • 验证新功能，可进行A/B Test进行功能验证，适合使用，快速验证。
  • 引入核心功能，保险起见，不推荐使用。
  • 引入非核心功能，适合运营类的功能，可以进行尝试。

快应用

发表于 2018-10-31

一、简介

快应用是基于手机硬件平台的新型应用形态，标准是由主流的手机厂商组成的快应用联盟联合制定，包括小米、华为、金立、魅族、vivo、OPPO、一加、中兴、联想、努比亚，来实现10亿部设备上无需安装、即点即用。

二、前置基础

1. 需要一些必备的Web基础
* HTML5、CSS3、JS基础
* ES6
2. 掌握基本的hap-toolkit的使用
* 初始化项目
* 编译
* 调试
* 打包
* ...
3. 掌握快应用IDE的使用

三、主要内容

1. 快应用与前端技术的异同点
1. 类似HTML语法，样式对CSS进行了一些裁剪和扩充，
2. 不支持浮动`float`、绝对定位`absolute`、相对定位`relative`等
3. 快应用的标签与HTML5标签含义不同    
    * 显示文本放置于`text`组件，不能放置于`div`组件
    * 基础组件中`a`和`text`组件支持内部嵌套组件，其它不支持
    * `a`的子组件`span`
    * `text`的子组件`a`和`span`
2. 样式书写
1. 盒模型：默认使用border-box模型，暂不支持content-box模型与box-sizing属性
    * `width`包括`padding`和`border`
    * `height`包括`padding`和`border`

2. 长度单位
    * px：与Web不一样，px是参考项目基准宽度，默认是750
    * %：参考父级
    * 计算公式：设计稿1px / 设计稿基准宽度 = 框架样式1px / 项目配置基准宽度

3. 选择器支持
    * id
    * class
    * tag
    * 并列
    * 后代

4. 样式引入
    * 内部
    * 外部
3. 快应用Flex布局
1. 主轴和交叉轴、容器和项目
2. 快应用容器标签仅为`div`、`list-item`、`tabs`
3. 容器和项目各自支持的属性(截止1020版本)
4. 容器属性
    * `flex-direction`（不支持 `*-reverse`）
    * `flex-wrap`
    * `flex-flow`（不支持）
    * `justify-content`（不支持`space-around`）
    * `aline-items`（不支持`baseline`）
    * `align-content`

5. 项目属性    
    * `order`（不支持）
    * `flex-gorw`
    * `flex-shrink`
    * `flex-basis`（和HTML5特性不一致）
    * `flex`（和HTML5特性不一致）
    * `align-self`

6. 快应用Flex布局与HTML5对比总结

  • 编译模式

编译脚本中的--release-debuginfo选项表示RelWithDebInfo模式来进行编译；

其他如--release选项表示版本追求了最大的性能优化，抛弃了调试信息；

--debug选项表示版本保留了全部的调试信息，而放弃了速度的优化；

--release-debuginfo选项表示版本结合了两者的优点，为了迅速的执行，代码经过了优化，但同时在编译期间也生成了调试用的数据和部分调试行，这让 debugger 有机会 catch 到出问题的代码空间是哪个部分。

不过，在实际开发中，这种有限的调试信息往往只能作为参考，如果确实需要寻找问题或加挂 debugger，还是需要使用 Debug 模式来编译。

  • 项目结构

Swift 标准库的源码大部分存放在swift/stdlib/public中，其中core文件夹下包括标准库中如String，Array等主要类型；SDK文件夹下是针对对应平台的代码、框架相关代码如 UIKit 和 Foundation等。
