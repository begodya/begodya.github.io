webpack 构建工程

发表于 2019-12-18

作用

webpack 可以看作是模块打包机：分析项目结构，找到 JavaScript 模块以及其它一些浏览器不能直接运行的扩展语言（如 TypeScript, stylus 等），将其转换并打包为合适的格式供浏览器使用。

工作方式

把项目当作一个整体，通过一个给定的主文件（如 index.js），webpack 将从这个文件开始查找与项目的所有依赖文件，使用 loader 进行处理，最后打包为一个或多个浏览器可以识别的 JavaScript 文件。

使用方法

安装

webpack 可通过 npm 进行安装，新建文件夹 webpack-go，在终端中进入该文件夹执行如下指令即可完成安装。

package.json 文件

package.json 是一个标准的 npm 说明文件，里面包含了丰富的信息，包含当前项目的依赖模块、自定义的脚本任务等。在终端中使用 npm init命令可自动创建 package.json 文件。

创建文件

在项目文件夹中创建文件夹src和build，src文件夹主要是用来存放原始数据和 JavaScript 模块，build文件夹主要用来存放之后供浏览器读取的文件（包括使用 webpack 打包生成的 js 文件）。

  • index.html – 根目录文件夹内，目的是引入打包后的 js 文件

  • greeter.js – src 文件夹内

  • main.js – src 文件夹内

webpack 配置文件

强大功能

  • source-map：打包时生成source maps，提供了一种对应编译文件和源文件的方法 ，使编译后代码可读性更高，且更易调试。

  • webpack-dev-server：构建本地服务器，让浏览器监听代码修改，并自动刷新显示修改后的结果。

  • Loaders：通过 loaders 实现对不同格式文件的转换处理，webpack 有能力调用外部的脚本或工具，loader 在打包构建过程中处理源文件（JSX，Scss，Less …），一次处理一个。

  • Babel：编译 JavaScript 的平台。

  • Plugin：拓展 webpack 功能，直接对整个构建过程起作用。

    • Html-webpack-plugin：依据一个简单的 index.html 模板，生成一个自动引用打包后JS文件的新 index.html。
    • Hot Module Replacement：热替换

vue-cli 及脚手架扩展

发表于 2019-11-27

准备工作

  1. vue-cli安装 npm install -g vue-cli
  2. webpack安装 npm install webpack -g
  3. 工程初始化 npm init
  4. Node 依赖安装 npm install

package.json

  1. 基础信息

    1. name： 项目名称
    2. version：项目版本
    3. Author：项目创建人
    4. private：是否私有

  2. 脚本 scripts：代理 webpack 原生命令，如函数封装，便于理解与调用

    1. “serve”: “vue-cli-service serve” = “webpack-dev-serve –inline –hot”
    2. “build”: “vue-cli-service build” = “webpack –config webpack.config.js -p”
    3. “lint”: “vue-cli-service lint”

  3. 项目依赖 dependencies：主要保存npm run build时构建项目的依赖包

    1. “vue”: “2.6.10”, 表示固定版本，只依赖2.6.10版本，之后有新版本不会更新
    2. “vue”: “~2.6.10”, 表示只安装2.6.X系列的最新版本
    3. “vue”: “^2.6.10”, 表示安装当前库的最新版本

  4. 开发依赖 devDependencies：开发依赖不会添加到生产环境中，只是在开发阶段辅助编译开发等功能

  5. 目标浏览器配置表 browserslist：表示是否兼容某些浏览器

    根据提供的目标浏览器的环境来，智能添加 css 前缀，js 的 polyfill 垫片，来兼容旧版本浏览器，而不是一股脑的添加。避免不必要的兼容代码，以提高代码的编译质量。> 根据提供的目标浏览器的环境来，智能添加css 前缀，js 的 polyfill 垫片，来兼容旧版本浏览器，而不是一股脑的添加。避免不必要的兼容代码，以提高代码的编译质量。

Vue.config.js

  1. configureWebpack：调整 webpack 配置最简单的方式
  2. chainWebpack：允许更细粒度的控制其内部配置

vue-cli-service 机制（仍需挖掘）

使用 vue 创建工程时，默认执行 npm run serve 就是 vue-cli-service serve，因为 vue-cli 3.0 安装时把 vue-cli-service 一并安装了，即执行了 npm install vue-cli-service --save-dev

文件入口：/node_modules/@vue/cli-service/bin/vue-cli-service.js

初始化配置：/node_modules/@vue/cli-service/lib/Service.js

webpack

  1. 核心原理
    1. 一切皆模块：将业务分割成更小的易于管理的片段，从而达到重复利用的目的
    2. 按需加载：使用特性来分割代码然后生成多个 bundle 文件，且异步加载部分代码以实现按需加载
  2. 核心概念
    1. 入口 entry：指示 webpack 应该使用哪个模块，来作为构建其内部依赖图的开始。
    2. 输出 output：告诉 webpack 在哪里输出它所创建的 bundles，以及如何命名这些文件，默认值为 ./dist。
    3. 加载器 loader： webpack 自身只理解 JavaScript，loader 可以将所有类型的文件转换为 webpack 能够处理的有效模块，然后就可以利用 webpack 的打包能力，进行处理。
    4. 插件 plugins：处理从打包优化和压缩，一直到重新定义环境中的变量。

命令解析

  1. 服务器部分

    1. webpack CLI tool：默认交互方式（已随 webpack 本身安装到本地）
    2. webpack-dev-server：node.js 服务器（需要开发者从 npm 自行安装）

  2. 参数部分

    1. webpack-dev-server --inline --hot --config webpack.config.dev.js

      • inline 开启热加载功能

      • hot 开启热替换功能

      • 当资源发生改变，以下三种方式都会生成新的 bundle，但是又有区别：

        // 1. 不会刷新浏览器

        $ webpack-dev-server

        // 2. 刷新浏览器

        $ webpack-dev-server –inline

        // 3. 重新加载改变的部分，HRM失败则刷新页面

        $ webpack-dev-server –inline –hot

大前端架构

发表于 2019-09-07

问题

随着现在终端类型越来越多，“一云多端”成为趋势。前后端就像一只八爪鱼，头部就是后端，触手就是各个前端。业务为了更快速、更广泛的触达到用户，多端已经必不可少，大前端会成为大趋势。

那么何为大前端，大前端之间是如何协作的？

目前大前端有两种范围所属：

1、Web + Node.js
2、App (iOS/Android) + Web + 小程序 + 快应用

第一种是在Node.js出现后，Web开发人员可以通过Node.js开发一部分业务服务，解放了后端人员的一部分时间；第二种是近几年终端类型多样化的规划；

App (iOS/Android) + Web (Node.js) + 小程序 + 快应用，我更偏向于两种的融合，以及未来的显示设备终端，所有面向用户的终端，更注重交互与体检，大前端是为了进行高效的开发，更好的维护，提高各方的能力输出，达到一次编写、多端运行的效果。

发表于 2019-03-25

BFC

以<div></div>为例，<div></div>被浏览器解析后会生成div元素并添加到DOM树中，CSS作用的对象不是DOM树，而是根据DOM树生成的Render树，BFC就是Render树的节点。

方法
  • float：不为none
  • position：不为relative、static
  • overflow：不为hidden
  • display：table-cell、table-caption、inline-block
结构
  • content box（必备）
  • padding box
  • border box
  • margin box
作用
  • 清除浮动
  • 自适应两栏布局
  • 防止margin合并
注意
  • CSS使用于Box，而不是作用于Element
  • JS无法操作Box
  • 同一个BFC的两个Box，margin会有重叠
误区
  • 通过width/height属性设置div元素的宽/高（错误）
  • CSS属性width/height作用于div元素所产生的Box，而不是Element本身
  • BFC有4个Box组成，那width/height到底作用于哪个Box？

IFC

因素
  • font-size
  • line-height
  • height
  • vertical-aligin
注意
  • inline-level box的高度是由字体字号大小决定的
  • inline-level box的宽度是由content+padding+border+margin决定的
  • inline-level box宽度大于父容器的宽度时会被拆分成多个inline-level box
  • 当达到拆分条件时，inline-level会被拆分成多个inline-level box并分布到多行中。
    • 当direction为ltr时，margin/border/padding-left只会作用于第一行inline-level box；margin/border/padding-right作用于最后一行inline-level box。
    • 当direction为rtl时，margin/border/padding-right只会作用于第一行inline-level box；margin/border/padding-left作用于最后一行inline-level box。

FFC

  • display: flex;
